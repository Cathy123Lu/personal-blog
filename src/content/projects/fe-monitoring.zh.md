## 背景

线上前端问题依赖用户反馈或后端日志间接发现，平均发现时间 40 分钟，严重影响用户体验和业务指标。

## 系统架构

```
Browser SDK → Kafka → 消费服务 → ClickHouse
                                      ↓
                              查询 API (Node.js)
                                      ↓
                              监控大盘 (Vue 3)
                                      ↓
                              告警规则引擎 → 钉钉/飞书
```

### Browser SDK 设计

轻量 SDK（gzip 后 4KB），通过 `PerformanceObserver` 采集 LCP/CLS/INP，通过 `window.onerror` + `unhandledrejection` 捕获 JS 错误，通过 XHR/Fetch 拦截采集接口异常。

上报策略：
- 优先使用 `sendBeacon`（页面卸载时不丢数据）
- 本地聚合后批量上报（每 5s 或满 20 条触发）
- 采样率可远程配置（高峰期自动降采样）

### 告警规则引擎

支持两类规则：
1. **阈值告警**：错误率 > 1% 持续 3 分钟触发
2. **环比告警**：与上周同时段相比上涨 > 50% 触发

规则配置存 MySQL，每分钟由 Node.js 定时任务拉取 ClickHouse 聚合数据做判断，命中则推送钉钉/飞书 webhook。

## 关键优化

**ClickHouse 查询优化**：将时间维度作为分区键，错误类型作为排序键，P99 查询从 4s 降到 800ms。

**误报抑制**：引入滑动窗口计数，单次毛刺不触发告警，连续 3 个窗口超阈值才告警，误报率从 15% 降到 2%。
